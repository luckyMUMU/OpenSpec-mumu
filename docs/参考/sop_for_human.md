## 1. SOP 概述

### 1.1 核心原则

SOP (Standard Operating Procedure) 是 AI Agent 执行任务的标准化操作流程。v2.12.0 版本采用 **Skill-first 架构**，以技能为核心组织原则。

**五大核心原则**：

| 原则 | 描述 |
|------|------|
| **Skill-first** | 所有操作通过 Skill 执行，Skill 是能力的唯一入口 |
| **渐进式披露** | 信息按 L1→L2→L3→L4 层级逐步展开 |
| **质量门禁** | 每个阶段完成后必须通过质量检查才能继续 |
| **路径分离** | 根据任务复杂度选择 Fast Path 或 Deep Path |
| **熔断保护** | 三击熔断机制防止无限循环 |

### 1.2 文档结构

```
sop/
├── AGENT_SOP.md          # 主入口文档
├── 01_concept_overview.md # 核心概念
├── 02_skill_matrix/      # Skill 矩阵
│   └── index.md
├── 03_workflow/          # 工作流规范
│   ├── index.md
│   ├── fast_path.md
│   ├── deep_path.md
│   └── three_strike_rule.md
```

### 1.3 Skill 总览

SOP v2.12.0 定义了 **17 个 Skill**，分为 4 大类别：

| 类别 | Skill 数量 | 职责范围 |
|------|-----------|----------|

| 路径与测试 (Path & Testing) | 5 | 路径选择、测试策略、执行控制 |

## 2. 渐进式披露机制

### 2.1 信息层级定义

渐进式披露 (Progressive Disclosure) 是 SOP 的核心设计模式，信息按层级逐步展开：

| 层级 | 名称 | 内容深度 | 使用场景 |
|------|------|----------|----------|
| **L1** | 概要层 | 核心概念、关键约束 | 快速理解、入口决策 |
| **L2** | 操作层 | 操作步骤、执行流程 | 日常执行、路径选择 |
| **L3** | 规范层 | 详细规范、边界条件 | 复杂场景、异常处理 |
| **L4** | 参考层 | 完整参考、边界案例 | 深度理解、问题排查 |

### 2.2 层级访问规则

```
L1 (概要层)
  └── 包含：核心原则、Skill 列表、关键约束
  └── 触发：任务开始时自动加载
  
L2 (操作层)
  └── 包含：工作流程、执行步骤、质量门禁
  └── 触发：路径确定后加载
  
L3 (规范层)
  └── 包含：详细规范、异常处理、边界条件
  └── 触发：进入 Deep Path 或遇到异常时加载
  
L4 (参考层)
  └── 包含：完整参考、历史案例、边界示例
  └── 触发：需要深度理解或问题排查时加载
```

### 2.3 信息密度控制

| 层级 | 最大行数 | 信息密度 | 示例 |
|------|----------|----------|------|
| L1 | 50 行 | 极简 | Skill 名称列表 |
| L2 | 200 行 | 精简 | 工作流程图 |
| L3 | 500 行 | 详细 | 完整规范文档 |
| L4 | 无限制 | 完整 | 所有参考文档 |

### 3.2 工作范围规则

**Skill 工作范围定义**：

```
工作范围 = {
    输入: [允许读取的文件/目录],
    输出: [允许写入的文件/目录],
    约束: [执行时的限制条件],
    依赖: [前置 Skill]
}
```

**范围边界规则**：

1. **输入隔离**：Skill 只能读取其工作范围内定义的输入
2. **输出限制**：Skill 只能写入其工作范围内定义的输出
3. **依赖顺序**：Skill 必须等待依赖项完成后才能执行
4. **约束强制**：所有约束条件必须在执行前验证

## 4. 核心约束

### 4.1 五大约束类别

| 约束类别 | 描述 | 强制级别 |
|----------|------|----------|
| **行为约束** | 定义允许/禁止的行为 | 强制 |
| **文件约束** | 定义文件访问权限 | 强制 |
| **状态约束** | 定义状态转换规则 | 强制 |
| **时间约束** | 定义超时和重试规则 | 推荐 |
| **质量约束** | 定义质量门禁标准 | 强制 |

### 4.2 禁止行为矩阵

| 行为 | Fast Path | Deep Path | TDD Path | 说明 |
|------|-----------|-----------|----------|------|
| 跳过质量门禁 | ❌ | ❌ | ❌ | 绝对禁止 |
| 绕过 Skill 执行 | ❌ | ❌ | ❌ | 绝对禁止 |
| 忽略熔断信号 | ❌ | ❌ | ❌ | 绝对禁止 |
| 修改核心约束 | ❌ | ❌ | ❌ | 绝对禁止 |
| 跳过测试验证 | ❌ | ❌ | ❌ | 绝对禁止 |
| 简化验收标准 | ❌ | ❌ | ❌ | 绝对禁止 |
| 并行目录执行 | ❌ | ✅ | ✅ | 仅 Deep Path 允许 |
| 回退到上一阶段 | ❌ | ✅ | ✅ | 仅 Deep Path 允许 |

### 4.3 文件访问约束

**允许访问的文件类型**：

```
允许读取:
├── .trae/              # Trae 配置目录

│   ├── rules/          # 规则目录
│   └── state/          # 状态目录
├── specs/              # 规格目录
├── src/                # 源代码目录
├── tests/              # 测试目录
└── docs/               # 文档目录

允许写入:
├── .trae/state/        # 状态文件
├── specs/              # 规格文件
├── src/                # 源代码
├── tests/              # 测试代码
└── docs/               # 文档
```

**禁止访问的文件**：

```
禁止访问:
├── .env                # 环境变量
├── .git/               # Git 目录
├── node_modules/       # 依赖目录
├── credentials/        # 凭证目录
└── secrets/            # 密钥目录
```

### 4.4 状态转换约束

**有效状态转换**：

```
pending → in_progress → completed
                ↓
            blocked → in_progress
                ↓
              failed → pending (重试)
```

**无效状态转换（禁止）**：

```
completed → in_progress  ❌ (已完成不能回退)
failed → completed       ❌ (失败不能直接完成)
pending → completed      ❌ (必须经过 in_progress)
```

## 5. 质量门禁机制

### 5.1 门禁定义

质量门禁 (Quality Gate) 是每个阶段完成后的质量检查点，必须通过才能进入下一阶段。

**门禁类型**：

| 门禁类型 | 检查内容 | 通过标准 | 阻塞级别 |
|----------|----------|----------|----------|
| **代码门禁** | 语法、类型、lint | 0 errors | 阻塞 |
| **测试门禁** | 单元测试、集成测试 | 100% 通过 | 阻塞 |
| **审查门禁** | 代码审查 | 无阻塞性问题 | 阻塞 |
| **文档门禁** | 文档完整性 | 必需文档存在 | 警告 |
| **验收门禁** | 需求满足度 | 100% 满足 | 阻塞 |

### 5.2 门禁执行流程

```
阶段完成
    │
    ▼
┌─────────────┐
│ 触发门禁检查 │
└─────────────┘
    │
    ▼
┌─────────────┐     失败
│ 门禁检查    │ ─────────────┐
└─────────────┘              │
    │ 通过                    ▼
    ▼                  ┌─────────────┐
┌─────────────┐        │ 记录失败原因 │
│ 记录通过状态 │        └─────────────┘
└─────────────┘              │
    │                        ▼
    ▼                  ┌─────────────┐
┌─────────────┐        │ 触发熔断检查 │
│ 进入下一阶段 │        └─────────────┘
└─────────────┘              │
                              ▼
                        ┌─────────────┐
                        │ 执行恢复策略 │
                        └─────────────┘
```

### 5.3 门禁检查清单

**代码门禁检查项**：

- [ ] 无语法错误
- [ ] 无类型错误
- [ ] Lint 检查通过
- [ ] 无安全漏洞
- [ ] 代码覆盖率达标

**测试门禁检查项**：

- [ ] 所有单元测试通过
- [ ] 所有集成测试通过
- [ ] 测试覆盖率 ≥ 80%
- [ ] 无跳过的测试
- [ ] 无 flaky 测试

**审查门禁检查项**：

- [ ] 代码审查完成
- [ ] 无阻塞性问题
- [ ] 所有评论已解决
- [ ] 设计文档已更新

## 6. 路径选择与工作流

### 6.1 路径类型

SOP 定义了三种执行路径：

| 路径 | 复杂度 | 特点 | 适用场景 |
|------|--------|------|----------|
| **Fast Path** | 低 | 单文件、无依赖、快速执行 | 简单修改、单文件任务 |
| **Deep Path** | 高 | 多文件、有依赖、目录并行 | 复杂功能、多模块任务 |
| **TDD Path** | 高 | 测试驱动、红绿循环 | 新功能开发、重构 |

### 6.2 路径选择决策树

```
任务输入
    │
    ▼
┌─────────────────┐
│ 评估任务复杂度  │
└─────────────────┘
    │
    ├── 单文件修改？
    │       │
    │       ├── 是 ──→ 无外部依赖？
    │       │               │
    │       │               ├── 是 ──→ Fast Path
    │       │               │
    │       │               └── 否 ──→ Deep Path
    │       │
    │       └── 否 ──→ Deep Path
    │
    └── 需要测试驱动？
            │
            ├── 是 ──→ TDD Path
            │
            └── 否 ──→ 继续评估
                    │
                    ├── 影响范围 ≤ 3 文件 ──→ Fast Path
                    │
                    └── 影响范围 > 3 文件 ──→ Deep Path
```

### 6.3 路径升级规则

**Fast Path → Deep Path 升级条件**：

1. 发现隐式依赖
2. 影响范围扩大
3. 连续失败 2 次
4. 用户明确要求

**Deep Path → TDD Path 升级条件**：

1. 新功能开发
2. 需要测试覆盖
3. 用户明确要求
4. 质量要求高

### 6.4 工作流状态机

```
                    ┌─────────────┐
                    │   START     │
                    └─────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │ path.selector│
                    │   路径选择   │
                    └─────────────┘
                          │
            ┌─────────────┼─────────────┐
            │             │             │
            ▼             ▼             ▼
    ┌───────────┐ ┌───────────┐ ┌───────────┐
    │Fast Path  │ │Deep Path  │ │TDD Path   │
    └───────────┘ └───────────┘ └───────────┘
            │             │             │
            └─────────────┼─────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │ 质量门禁检查 │
                    └─────────────┘
                          │
                    ┌─────┴─────┐
                    │           │
                通过 ▼           ▼ 失败
            ┌───────────┐ ┌───────────┐
            │  COMPLETE │ │  RECOVERY │
            └───────────┘ └───────────┘
```

## 7. 快速路径 (Fast Path)

### 7.1 Fast Path 定义

Fast Path 是为简单任务设计的快速执行路径，特点是：

- **单文件操作**：一次只修改一个文件
- **无外部依赖**：不需要其他文件的支持
- **快速执行**：跳过复杂的规划阶段
- **简化门禁**：只执行必要的检查

### 7.2 Fast Path 执行流程

```
┌─────────────────┐
│   任务输入      │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  路径选择确认   │
│  (path.selector)│
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  快速分析任务   │
│  (L1 信息加载)  │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  直接执行修改   │
│(path.fast_executor)│
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  简化质量检查   │
└─────────────────┘
        │
    ┌───┴───┐
    │       │
通过 ▼       ▼ 失败
┌───────┐ ┌───────────┐
│ 完成  │ │ 升级到    │
│       │ │ Deep Path │
└───────┘ └───────────┘
```

### 7.3 Fast Path 适用条件

**必须同时满足以下条件**：

| 条件 | 描述 | 验证方法 |
|------|------|----------|
| 单文件 | 只修改一个文件 | 文件计数 |
| 无依赖 | 不依赖其他文件修改 | 依赖分析 |
| 低风险 | 修改不会破坏现有功能 | 影响分析 |
| 明确需求 | 需求清晰无歧义 | 需求检查 |

### 7.4 Fast Path 升级触发

**自动升级到 Deep Path 的条件**：

1. **发现隐式依赖**
   - 执行过程中发现需要修改其他文件
   - 依赖分析发现跨文件影响

2. **执行失败**
   - 连续 2 次执行失败
   - 质量门禁检查失败

3. **复杂度超标**
   - 修改行数 > 100 行
   - 涉及多个函数/类

4. **用户请求**
   - 用户明确要求使用 Deep Path
   - 用户要求更详细的规划

### 7.5 Fast Path 时间限制

| 阶段 | 时间限制 | 超时处理 |
|------|----------|----------|
| 分析 | 30 秒 | 升级到 Deep Path |
| 执行 | 5 分钟 | 升级到 Deep Path |
| 检查 | 1 分钟 | 报告失败 |

## 8. 深度路径 (Deep Path)

### 8.1 Deep Path 定义

Deep Path 是为复杂任务设计的完整执行路径，特点是：

- **多文件操作**：可以同时修改多个文件
- **目录并行执行**：按目录分组并行处理
- **完整规划**：详细的任务分解和规划
- **完整门禁**：执行所有质量检查

```
        │
        ▼
┌─────────────────┐
│  路径选择确认   │
│  (path.selector)│
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  目录规划       │
│(directory_planner)│
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  需求分析       │
│(requirements.analyzer)│
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  规格编写       │
│(spec_writer)    │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  任务分解       │
│(task_decomposer)│
└─────────────────┘
        │
        ▼
┌─────────────────┐

│(deep_executor)  │
└─────────────────┘
        │

└─────────────────┘
        │
    ┌───┴───┐
    │       │
通过 ▼       ▼ 失败
┌───────┐ ┌───────────┐
│ 完成  │ │ 熔断检查  │
└───────┘ └───────────┘
```

### 8.3 目录并行执行机制

**目录分组规则**：

```
项目结构:
├── src/
│   ├── core/       ← 目录组 1
│   ├── utils/      ← 目录组 2
│   └── api/        ← 目录组 3
├── tests/
│   ├── unit/       ← 目录组 4
│   └── integration/← 目录组 5
└── docs/           ← 目录组 6

执行顺序:
并行组 1: [core/, utils/]  (无依赖)
    ↓
并行组 2: [api/]           (依赖 core)
    ↓
并行组 3: [tests/]         (依赖 src)
    ↓
串行组: [docs/]            (最后执行)
```

**并行执行规则**：

1. **依赖分析**：分析目录间的依赖关系
2. **拓扑排序**：按依赖关系排序
3. **分组并行**：无依赖的目录并行执行
4. **顺序保证**：有依赖的目录按顺序执行

### 8.4 Deep Path 状态管理

**状态快照机制**：

```
状态快照 = {
```

- 每个阶段完成后保存状态快照
- 中断后可以从最后一个快照恢复
- 状态文件位置：`.trae/state/snapshots/`

## 9. TDD 深度路径

### 9.1 TDD Path 定义

TDD (Test-Driven Development) Path 是测试驱动的深度路径，特点是：

- **测试优先**：先写测试，再写实现
- **红绿循环**：测试失败→实现→测试通过
- **小步迭代**：每次只实现一个小功能
- **持续重构**：在绿色状态下重构

### 9.2 TDD 执行流程

```
┌─────────────────┐
│   任务输入      │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  路径选择确认   │
│  (TDD 模式)     │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  编写测试用例   │
│  (RED 阶段)     │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  运行测试       │
│  (确认失败)     │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  编写最小实现   │
│  (GREEN 阶段)   │
└─────────────────┘
        │
        ▼
┌─────────────────┐
│  运行测试       │
│  (确认通过)     │
└─────────────────┘
        │
    ┌───┴───┐
    │       │
通过 ▼       ▼ 失败
┌───────┐ ┌───────────┐
│ 重构  │ │ 修复实现  │
│(REFACTOR)│ │(返回GREEN)│
└───────┘ └───────────┘
    │
    ▼
┌─────────────────┐
│  下一个测试用例 │
│  (循环)         │
└─────────────────┘
```

### 9.3 红-绿-重构循环

**RED 阶段（测试失败）**：

```
1. 理解需求
2. 编写失败的测试
3. 确认测试失败（预期原因）
4. 记录测试用例
```

**GREEN 阶段（测试通过）**：

```
1. 编写最小实现
2. 运行测试
3. 确认测试通过
4. 不进行任何重构
```

**REFACTOR 阶段（重构）**：

```
1. 确认测试通过
2. 识别代码异味
3. 小步重构
4. 每步后运行测试
5. 确认测试仍然通过
```

### 9.4 TDD 质量门禁

**每个循环的门禁检查**：

| 检查项 | RED | GREEN | REFACTOR |
|--------|-----|-------|----------|
| 测试存在 | ✅ | ✅ | ✅ |
| 测试失败 | ✅ | ❌ | ❌ |
| 测试通过 | ❌ | ✅ | ✅ |
| 代码质量 | - | - | ✅ |
| 覆盖率 | - | - | ✅ |

### 9.5 TDD 时间限制

| 阶段 | 时间限制 | 说明 |
|------|----------|------|
| RED | 5 分钟 | 编写单个测试 |
| GREEN | 10 分钟 | 实现最小代码 |
| REFACTOR | 10 分钟 | 每次重构 |
| 单循环 | 25 分钟 | 完整循环 |

## 10. 三击熔断规则

### 10.1 熔断机制定义

三击熔断 (Three-Strike Rule) 是防止无限循环的保护机制：

- **触发条件**：同一任务连续失败 3 次
- **熔断效果**：停止自动执行，请求人工干预
- **恢复机制**：提供恢复选项，允许重新尝试

### 10.2 熔断计数规则

```
熔断计数器:
├── strike_count: 当前失败次数 (0-3)
├── last_error: 最后一次错误信息
├── error_history: 错误历史记录
└── reset_condition: 重置条件

计数规则:
├── 连续失败: strike_count++
├── 成功执行: strike_count = 0
├── 用户干预: strike_count = 0
└── 达到 3 次: 触发熔断
```

### 10.3 熔断触发流程

```
执行失败
    │
    ▼
┌─────────────────┐
│ strike_count++  │
└─────────────────┘
    │
    ▼
┌─────────────────┐
│ strike_count ≥ 3?│
└─────────────────┘
    │
    ├── 否 ──→ 记录错误，重试
    │
    └── 是 ──→ 触发熔断
                    │
                    ▼
            ┌─────────────────┐
            │ 停止自动执行    │
            └─────────────────┘
                    │
                    ▼
            ┌─────────────────┐
            │ 生成错误报告    │
            └─────────────────┘
                    │
                    ▼
            ┌─────────────────┐
            │ 请求人工干预    │
            └─────────────────┘
```

### 10.4 熔断恢复选项

**熔断后可选择的恢复选项**：

| 选项 | 描述 | 效果 |
|------|------|------|
| **重试** | 使用相同参数重试 | strike_count = 0 |
| **降级** | 降级到更简单的路径 | strike_count = 0 |
| **跳过** | 跳过当前任务 | 记录跳过原因 |
| **人工** | 请求人工处理 | 等待人工完成 |
| **终止** | 终止整个流程 | 结束执行 |

### 10.5 熔断报告格式

```markdown
## 熔断报告

**触发时间**: ISO-8601 时间戳
**任务 ID**: task_xxx
**路径类型**: Fast/Deep/TDD

### 错误历史

| 次数 | 时间 | 错误类型 | 错误信息 |
|------|------|----------|----------|
| 1 | ... | ... | ... |
| 2 | ... | ... | ... |
| 3 | ... | ... | ... |

### 建议操作

1. [重试] - 使用相同参数重试
2. [降级] - 切换到更简单的路径
3. [跳过] - 跳过当前任务
4. [人工] - 请求人工处理
5. [终止] - 终止整个流程
```

## 11. 目录调度状态机

### 11.1 状态机定义

目录调度状态机管理多目录并行执行的状态转换：

```
状态定义:
├── PENDING: 等待执行
├── READY: 准备就绪（依赖已满足）
├── IN_PROGRESS: 执行中
├── BLOCKED: 被阻塞（依赖未满足）
├── COMPLETED: 已完成
└── FAILED: 执行失败
```

### 11.2 状态转换图

```
                    ┌─────────┐
                    │ PENDING │
                    └─────────┘
                         │
                         │ 依赖检查
                         ▼
              ┌──────────────────┐
              │ 依赖满足?        │
              └──────────────────┘
                    │       │
                是  │       │ 否
                    ▼       ▼
              ┌─────────┐ ┌─────────┐
              │  READY  │ │ BLOCKED │
              └─────────┘ └─────────┘
                    │           │
                    │ 调度      │ 依赖满足
                    ▼           │
              ┌─────────┐      │
              │IN_PROGRESS│◄────┘
              └─────────┘
                    │
            ┌───────┴───────┐
            │               │
        成功 ▼               ▼ 失败
      ┌───────────┐   ┌───────────┐
      │ COMPLETED │   │  FAILED   │
      └───────────┘   └───────────┘
```

### 11.3 状态转换规则

| 当前状态 | 触发事件 | 目标状态 | 条件 |
|----------|----------|----------|------|
| PENDING | 依赖检查 | READY | 所有依赖已完成 |
| PENDING | 依赖检查 | BLOCKED | 存在未完成依赖 |
| BLOCKED | 依赖完成 | READY | 所有依赖已完成 |
| READY | 调度执行 | IN_PROGRESS | 资源可用 |
| IN_PROGRESS | 执行成功 | COMPLETED | - |
| IN_PROGRESS | 执行失败 | FAILED | - |
| FAILED | 重试 | READY | 重试次数 < 3 |

### 11.4 并行调度算法

```
调度算法:
1. 收集所有 PENDING 状态的目录
2. 检查每个目录的依赖状态
3. 将依赖满足的目录标记为 READY
4. 从 READY 目录中选择执行
5. 并行执行选中的目录
6. 更新状态，重复步骤 1

选择策略:
├── 优先级优先: 按优先级排序
├── 依赖数优先: 依赖数多的优先（减少阻塞）
└── 公平调度: 轮询选择
```

### 11.5 状态持久化

**状态文件格式**：

```json
{
  "version": "2.9.0",
  "timestamp": "2024-01-15T10:30:00Z",
  "directories": {
    "src/core": {
      "status": "COMPLETED",
      "started_at": "2024-01-15T10:00:00Z",
      "completed_at": "2024-01-15T10:15:00Z",
      "dependencies": []
    },
    "src/api": {
      "status": "IN_PROGRESS",
      "started_at": "2024-01-15T10:15:00Z",
      "dependencies": ["src/core"]
    },
    "src/utils": {
      "status": "READY",
      "dependencies": []
    }
  }
}
```
