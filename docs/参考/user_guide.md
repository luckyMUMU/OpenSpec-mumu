# OpenSpec 用户指南

> **面向 AGENT 的 SOP v3.0.0 使用指南** - 指导如何使用 Spec-First SOP 进行复杂项目编程

---

## 一、快速入门

### 1.1 什么是 OpenSpec？

OpenSpec v3.0.0 是一套 AI 辅助软件工程的标准化流程体系。核心思想是：

- **规范是第一性产物**：Spec-First 架构，规范是系统的"宪法"
- **Skill 是规范的执行工具**：Skill 是规范的"翻译器"，负责执行规范
- **契约驱动执行**：阶段间通过契约文件（YAML）通信，不共享状态
- **可审计闭环**：所有产出必须声明来源与依赖

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **Spec-First** | 规范优先，规范是第一性产物 |
| **P0-P3 规范** | 四级规范体系（工程宪章/系统规范/模块规范/实现规范） |
| **5 阶段工作流** | Stage 0-4（权重选择/设计/实现/交付/归档） |
| **契约文件** | YAML 格式，定义前置/后置条件、不变式 |
| **Skill 分类** | 4 类 Skill（编排/规范/实现/验证） |

### 1.3 目录结构

```text
project/
├── docs/
│   ├── 01_requirements/         # L1: 需求文档
│   ├── 02_logical_workflow/     # L2: 架构设计
│   ├── 03_technical_spec/       # L3: 技术规格
│   └── 04_context_reference/    # L4: 决策记录
├── src/                         # 源代码
│   └── **/design.md             # 目录级实现设计
├── tests/                       # 测试代码
├── sop/                         # SOP v3.0.0 规范
│   ├── 01_constitution/         # P0 级工程宪章
│   ├── 02_specifications/       # P1-P2 级系统规范
│   ├── 03_workflow/             # 5 阶段工作流
│   ├── 04_skills/               # 4 类 Skill
│   ├── 05_constraints/          # P0-P3 约束体系
│   ├── 06_templates/            # 模板体系
│   └── 07_reference/            # 参考资料
└── .trae/specs/                 # 任务执行期临时规范
```

## 二、工作流程

### 2.1 5 阶段工作流

SOP v3.0.0 使用 5 阶段契约驱动工作流：

**Stage 0 - 权重选择**：
- 活动：任务分诊、路径选择、复杂度评估
- 输出：路径决策（Deep/Fast）
- 契约：`stage-0-contract.yaml`

**Stage 1 - 设计**：
- 活动：需求分析、架构设计、API 定义
- 输出：design.md、API 契约
- 契约：`stage-1-contract.yaml`

**Stage 2 - 实现**：
- 活动：代码实现、测试编写
- 输出：源代码、测试代码
- 契约：`stage-2-contract.yaml`

**Stage 3 - 交付**：
- 活动：验收测试、文档同步
- 输出：验收报告、文档更新
- 契约：`stage-3-contract.yaml`

**Stage 4 - 归档**：
- 活动：知识沉淀、状态清理
- 输出：归档文件、经验总结
- 契约：`stage-4-contract.yaml`

### 2.2 路径选择

**Stage 0** 阶段进行路径选择：

#### 快速路径（Fast Path）

**适用条件**（必须同时满足）：
- 单文件修改
- <30 行代码变更
- 无逻辑变更

**流程**：
```
Stage 0 → Stage 2 → Stage 3 → Stage 4
        (跳过 Stage 1)
```

#### 深度路径（Deep Path）

**适用条件**（满足任一）：
- 跨文件修改
- 新功能开发
- 重构任务
- API 变更
- 架构调整

**流程**：
```
Stage 0 → Stage 1 → Stage 2 → Stage 3 → Stage 4
```

### 2.3 质量门控机制

每个阶段完成后必须执行门控检查（P0 级约束）：

| 阶段 | 门控检查项 | 通过条件 | 失败处理 |
|------|-----------|----------|----------|
| Stage 0 | 路径选择合理、复杂度评估准确 | 全部通过 | 重新评估路径 |
| Stage 1 | 设计文档完整、接口定义清晰、无架构冲突 | 全部通过 | 返回设计修正 |
| Stage 2 | 代码规范、测试通过、契约满足 | 全部通过 | 返回实现修正 |
| Stage 3 | 验收通过、文档同步、无遗留问题 | 全部通过 | 返回交付修正 |
| Stage 4 | 归档完整、状态清理、知识沉淀 | 全部通过 | 返回归档修正 |

**门控失败约束（P0 级）**：
- 门控失败触发熔断机制
- 每次失败必须记录到契约文件
- 用户决策：修复后重试、回滚到上一阶段、终止任务
- 与三错即停机制独立（门控失败不累计到三错计数）

### 2.3 目录维度并行执行

当任务涉及多个包含 `design.md` 的目录时：

1. **扫描**：识别所有包含 `design.md` 的目录
2. **调度**：根据目录深度构建依赖关系
3. **执行**：按深度自底向上执行（同深度无依赖可并行）
4. **依赖处理**：跨目录依赖进入等待状态

**目录调度状态机**：

```
[DIR_WAITING_DEP] ←── 依赖未就绪 ──┐
       │                          │
       ↓ 依赖就绪（自动触发）       │
[DIR_WORKING] ──→ 处理完成 ──→ [DIR_COMPLETED]
       │                          │
       ↓ 处理失败                 │
[DIR_FAILED] ─────────────────────┘
```

**调度状态保存格式**（JSON）：
```json
{
  "version": "1.0",
  "timestamp": "2026-02-24T10:30:00Z",
  "directories": [
    {
      "path": "src/module-a",
      "state": "DIR_COMPLETED",
      "design_md": "src/module-a/design.md",
      "completed_at": "2026-02-24T10:00:00Z"
    }
  ],
  "current_batch": 1,
  "total_batches": 3
}
```
保存位置：`.trae/scheduler_state.json`

### 2.4 Spec 驱动开发

使用 Spec 进行任务管理：

```text
.trae/specs/<change-id>/
├── spec.md      # 规范定义（Why/What/Impact）
├── tasks.md     # 任务列表（可验证的工作项）
└── checklist.md # 检查清单（验证点）
```

**Spec 生命周期**：
- **创建**：任务开始时创建 spec.md
- **执行**：按 tasks.md 执行，使用 checklist.md 验证
- **归档**：重要任务完成后归档到 `docs/04_context_reference/archived_specs/`
- **清理**：简单任务完成后直接删除

## 三、Skill 分类体系

### 3.1 4 类 Skill

SOP v3.0.0 将 Skill 分为 4 大类别：

| 类别 | 职责 | 主要输出 |
|------|------|----------|
| **编排类 (Orchestration)** | 任务调度、状态管理、流程控制 | Stage 契约、状态报告 |
| **规范类 (Specification)** | 需求分析、架构设计、任务分解 | L1-L2 文档、design.md |
| **实现类 (Implementation)** | 代码实现、测试编写 | 源代码、测试代码 |
| **验证类 (Verification)** | 质量审查、验收确认 | 审查报告、验收报告 |

### 3.2 Skill 执行规范

每个 Skill 执行必须遵循：

1. **读取契约**：从 Stage 契约文件获取前置条件
2. **执行活动**：按规范定义执行
3. **验证契约**：确认满足后置条件
4. **更新状态**：更新契约文件状态

### 3.3 停止点处理

当遇到停止点时：

| 停止点 | 处理方式 |
|--------|----------|
| `[USER_DECISION]` | 等待用户决策，记录决策到契约文件 |
| `[DIR_WAITING_DEP]` | 等待依赖目录完成 |
| `[BLOCKED]` | 记录阻塞原因，等待解除 |
| `[GATE_FAILED]` | 门控失败，等待用户决策（修复/回滚/终止） |
| `[THREE_STRIKES]` | 三错熔断，等待用户决策 |

## 四、最佳实践

### 4.1 规范先行原则

1. **阅读规范**：执行前阅读相关 P0-P3 规范
2. **遵守契约**：严格按 Stage 契约执行
3. **记录决策**：重要决策记录到契约文件

### 4.2 任务划分原则

| 任务类型 | 划分方式 |
|----------|----------|
| 单目录任务 | 对应单个 design.md |
| 跨目录任务 | 拆分为多个子任务 |
| 复杂任务 | 使用 Spec 三件套管理 |

### 4.3 执行顺序原则

1. **自底向上**：从最深目录开始执行
2. **并行执行**：同深度无依赖的任务可并行
3. **依赖等待**：有依赖的任务等待依赖完成后执行

### 4.4 文档规范

1. **来源声明**：所有分析文档必须声明来源
2. **依赖声明**：设计文档必须声明依赖
3. **决策记录**：重要决策必须记录到契约文件

## 五、常见问题

### 5.1 如何选择工作流路径？

根据任务复杂度判断：
- **快速路径**：单文件、小变更、无逻辑变化
- **深度路径**：新功能、重构、跨文件修改

### 5.2 如何处理跨目录依赖？

1. 识别依赖关系
2. 按深度排序
3. 自底向上执行
4. 依赖目录完成后唤醒等待任务

### 5.3 Spec 产物如何处理？

- **重要任务**：归档到 `docs/04_context_reference/archived_specs/`
- **简单任务**：完成后直接删除
- **设计内容**：迁移到 `design.md` 或 ADR

### 5.4 版本不一致如何处理？

1. 确认 CHANGELOG 基准版本
2. 批量更新所有文件版本号
3. 更新 updated 日期
4. 验证一致性

### 5.5 遇到停止点怎么办？

1. 记录当前状态
2. 更新契约文件
3. 等待外部输入
4. 恢复执行时读取契约文件

### 5.6 如何处理设计冲突？

当系统检测到设计冲突时，按以下步骤处理：

1. **确认冲突级别**：根据影响范围确定优先级（P0-P3）
2. **分析冲突原因**：查看冲突检测报告，理解冲突来源
3. **制定解决方案**：
   - P0 级冲突：立即停止，必须解决后继续
   - P1 级冲突：暂停当前任务，优先处理
   - P2/P3 级冲突：记录到待办，批量或后续处理
4. **用户确认**：将解决方案提交用户确认
5. **更新文档**：根据决策更新相关文档

### 5.7 用户决策如何记录？

用户决策按分级原则记录到不同位置：

| 决策类型 | 记录位置 | 示例 |
|----------|----------|------|
| 架构级决策 | ADR | 技术栈选型、核心架构模式 |
| 设计级决策 | design.md | 模块接口定义、数据结构设计 |
| 实现级决策 | spec.md | 具体实现方案、临时技术选择 |
| 操作级决策 | 契约文件 | 日常配置、命名约定 |

记录内容包括：
- 决策背景与原因
- 可选方案及对比
- 最终决策及理由
- 影响范围评估

### 5.8 ADR 更新流程是什么？

ADR 更新遵循以下流程：

1. **触发更新**：
   - 用户做出架构级决策
   - 检测到与现有 ADR 冲突
   - 技术栈或核心模式变更

2. **创建草稿**：
   - 基于决策内容生成 ADR 草稿
   - 填写决策背景、方案对比、最终选择

3. **用户确认**：
   - 提交用户审核 ADR 内容
   - 根据反馈修改完善

4. **正式归档**：
   - 确认后保存到 `docs/04_context_reference/adr/`
   - 更新相关索引文档

5. **同步更新**：
   - 通知受影响的设计文档
   - 更新约束矩阵（如有需要）
