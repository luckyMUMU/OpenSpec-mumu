# P1 级约束

> **约束强度**: 跨模块约束，警告可接受

---

## 概述

P1 级约束定义系统级质量要求，跨模块约束，变更需技术负责人审批。

---

## 性能约束

### P1-PERF-001: API 响应时间

| 属性 | 值 |
|------|-----|
| **描述** | API 响应时间必须 < 500ms（P95） |
| **验证方法** | 性能测试（k6、JMeter） |
| **违反处理** | 警告，建议优化 |
| **例外流程** | 复杂查询可放宽至 2s（需审批） |

### P1-PERF-002: 数据库查询优化

| 属性 | 值 |
|------|-----|
| **描述** | 禁止 N+1 查询问题 |
| **验证方法** | 慢查询日志、APM 工具 |
| **违反处理** | 警告，建议优化 |
| **例外流程** | 批量操作可例外（需审批） |

---

## 可用性约束

### P1-AVAIL-001: 服务可用性

| 属性 | 值 |
|------|-----|
| **描述** | 服务可用性必须 >= 99.9% |
| **验证方法** | 监控系统（Prometheus、Grafana） |
| **违反处理** | 警告，需要改进 |
| **例外流程** | 计划维护期间可例外 |

### P1-AVAIL-002: 故障恢复时间

| 属性 | 值 |
|------|-----|
| **描述** | 故障恢复时间必须 < 5min |
| **验证方法** | 故障演练（Chaos Engineering） |
| **违反处理** | 警告，需要改进 |
| **例外流程** | 重大故障可放宽（需审批） |

## 技术约束

### P1-TECH-001: 优先使用项目已有库

| 属性 | 值 |
|------|-----|
| **描述** | 新功能优先使用项目已有技术栈 |
| **验证方法** | 依赖审查 |
| **违反处理** | 警告，建议复用 |
| **例外流程** | 无替代方案时可引入（需 ADR） |

### P1-TECH-002: 禁止重复功能库

| 属性 | 值 |
|------|-----|
| **描述** | 禁止引入功能重复的库 |
| **验证方法** | 功能对比审查 |
| **违反处理** | 警告，建议移除 |
| **例外流程** | 性能优势明显时可例外（需审批） |

## 接口约束

### P1-API-001: API 版本管理

| 属性 | 值 |

| **描述** | API 必须有版本号，遵循 SemVer |
| **验证方法** | API 文档审查 |
| **违反处理** | 警告，建议添加版本号 |
| **例外流程** | 内部 API 可例外（需审批） |

### P1-API-002: API 向后兼容

| 属性 | 值 |
|------|-----|
| **描述** | API 变更必须保持向后兼容 |
| **验证方法** | API 契约测试 |
| **违反处理** | 警告，建议兼容处理 |
| **例外流程** | 主版本升级时可例外（需审批） |

## 需求约束

### P1-REQ-001: 需求澄清完整性

| 属性 | 值 |
|------|-----|
| **描述** | 阶段 1 必须完成多轮次多维度提问，记录用户真实意图 |
| **验证方法** | 检查 `specs/{name}-clarification.md` 文档完整性 |
| **违反处理** | 警告，建议补充澄清 |
| **例外流程** | 快速路径可跳过（需审批） |

**必需维度**：
- 业务背景与目标
- 长期目标与短期实现
- 边界与约束
- 风险与假设

### P1-REQ-002: 长期短期区分

| 属性 | 值 |
|------|-----|
| **描述** | 需求必须明确区分长期目标与短期实现 |
| **验证方法** | 检查澄清记录中是否包含"长期愿景"和"短期目标" |
| **违反处理** | 警告，建议补充区分 |
| **例外流程** | 简单需求可例外（需审批） |
