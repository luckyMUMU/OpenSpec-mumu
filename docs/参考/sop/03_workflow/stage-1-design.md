# 阶段 1：理解与设计

> **阶段目标**: 理解需求，完成设计（BDD 场景 + DDD 战术映射）

---

## 阶段概述

阶段 1 负责深入理解需求，完成架构设计和实现设计，产出设计文档和契约。

## 需求澄清规范（多轮次多维度提问）

### 第一轮提问：业务背景与目标

**目标**：理解需求的业务背景和核心目标

| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 业务背景 | "这个需求要解决什么业务问题？" | 理解问题本质 |
| 目标用户 | "谁会使用这个功能？他们的核心诉求是什么？" | 明确用户画像 |
| 成功标准 | "如何衡量这个功能的成功？" | 定义验收标准 |
| 紧急程度 | "这个需求的优先级如何？是否有截止日期？" | 评估优先级 |

### 第二轮提问：长期目标与短期实现

**目标**：区分长期愿景与短期可交付成果

| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 长期愿景 | "这个功能最终要达到什么状态？" | 理解终极目标 |
| 短期目标 | "第一期需要交付哪些核心能力？" | 定义 MVP 范围 |
| 演进路径 | "从短期到长期，计划的演进路径是什么？" | 规划迭代路线 |
| 技术债务 | "短期实现是否可以接受技术债务？如何偿还？" | 评估技术风险 |

### 第三轮提问：边界与约束

**目标**：明确功能边界和约束条件

| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 功能边界 | "这个功能不包含什么？" | 避免范围蔓延 |
| 技术约束 | "是否有技术栈、性能、安全方面的约束？" | 识别技术限制 |
| 兼容性 | "是否需要兼容现有系统？有哪些依赖？" | 评估集成影响 |
| 资源限制 | "团队规模、时间预算、技术能力如何？" | 评估可行性 |

### 第四轮提问：风险与假设

**目标**：识别潜在风险和关键假设

| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 风险识别 | "这个需求最大的风险是什么？" | 提前规避风险 |
| 关键假设 | "我们做了哪些假设？这些假设是否已验证？" | 暴露隐藏假设 |
| 失败场景 | "如果失败，可能的失败模式是什么？" | 设计降级方案 |
| 回滚策略 | "如果上线后出现问题，如何回滚？" | 制定应急预案 |

## BDD 场景描述规范

### Gherkin 语法

```gherkin
Feature: [功能名称]
  As a [角色]
  I want to [功能]
  So that [价值]

  Scenario: [场景名称]
    Given [前置条件]
    When [触发动作]
    Then [预期结果]
    And [额外结果]

  Scenario: [异常场景名称]
    Given [前置条件]
    When [触发动作]
    Then [预期异常结果]
```

### 场景覆盖要求

| 场景类型 | 覆盖率要求 | 描述 |
|----------|-----------|------|
| 正常流程 | 100% | 所有可能的正常路径 |
| 异常流程 | 100% | 所有可能的异常路径 |
| 边界条件 | 关键边界 | 关键的边界值测试 |

## DDD 战术映射规范

### 聚合根

| 规则 | 描述 |
|------|------|
| 唯一入口 | 聚合根是访问聚合内实体的唯一入口 |
| 边界保护 | 聚合外不能直接修改聚合内实体 |
| 事务边界 | 一个聚合 = 一个事务 |

### 值对象

| 规则 | 描述 |
|------|------|
| 不可变 | 值对象创建后不可修改 |
| 相等性 | 值对象通过属性值判断相等 |
| 自验证 | 值对象构造时验证自身有效性 |

### 领域服务

| 规则 | 描述 |
|------|------|
| 无状态 | 领域服务不应持有状态 |
| 跨实体 | 封装跨实体的业务逻辑 |
| 领域语言 | 方法名应使用领域语言 |

### 仓储模式

| 规则 | 描述 |
|------|------|
| 接口在领域层 | 仓储接口定义在领域层 |
| 实现在基础设施层 | 仓储实现在基础设施层 |
| 隐藏实现细节 | 不暴露数据库细节 |

## 质量门控

| 检查项 | 通过条件 | 失败处理 |
|--------|----------|----------|
| 需求澄清完成 | 完成 4 轮提问，记录完整 | 返回需求澄清 |
| 长期短期区分 | 明确长期目标与短期实现 | 返回需求澄清 |
| 用户意图确认 | 用户确认需求澄清记录 | 返回需求澄清 |
| 需求边界清晰 | 功能范围明确，无歧义 | 返回需求分析 |
| 技术方案对齐 | 技术选型合理，与现有系统兼容 | 返回架构设计 |
| 验收标准具体 | 每个功能都有明确的验收条件 | 返回需求分析 |
| 关键假设确认 | 所有关键假设已与用户确认 | 返回需求分析 |
| 架构图清晰 | 架构图完整，模块边界明确 | 返回架构设计 |
| 接口定义完整 | 所有接口都有明确的输入输出定义 | 返回实现设计 |
| 与现有系统无冲突 | 不与现有架构/接口冲突 | 返回架构设计 |
| 设计可行 | 技术方案可实现 | 返回实现设计 |

## 状态定义

| 状态 | 触发者 | 含义 | 后续动作 |
|------|--------|------|----------|
| `[STAGE_1_STARTED]` | 阶段 0 通过 | 阶段 1 开始 | 执行需求澄清 |
| `[STAGE_1_CLARIFYING]` | 需求澄清 | 正在进行多轮提问 | 等待澄清完成 |
| `[STAGE_1_ANALYZING]` | 需求分析 | 正在分析需求 | 等待分析完成 |
| `[STAGE_1_DESIGNING]` | 设计阶段 | 正在设计 | 等待设计完成 |
| `[STAGE_1_REVIEWING]` | 设计审查 | 正在审查设计 | 等待审查完成 |
| `[STAGE_1_WAITING_CONFIRM]` | 审查完成 | 等待用户确认 | 用户确认后进入阶段 2 |
| `[STAGE_1_PASSED]` | 用户确认 | 阶段 1 通过 | 进入阶段 2 |
| `[STAGE_1_FAILED]` | 审查失败 | 设计审查未通过 | 返回设计修正 |
