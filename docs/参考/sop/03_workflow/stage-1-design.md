## BDD 场景描述规范

### Gherkin 语法

```gherkin
Feature: [功能名称]
  As a [角色]
  I want to [功能]
  So that [价值]

  Scenario: [场景名称]
    Given [前置条件]
    When [触发动作]
    Then [预期结果]
    And [额外结果]

  Scenario: [异常场景名称]
    Given [前置条件]
    When [触发动作]
    Then [预期异常结果]
```

### 场景覆盖要求

| 场景类型 | 覆盖率要求 | 描述 |
|----------|-----------|------|
| 正常流程 | 100% | 所有可能的正常路径 |
| 异常流程 | 100% | 所有可能的异常路径 |
| 边界条件 | 关键边界 | 关键的边界值测试 |

## DDD 战术映射规范

### 聚合根

| 规则 | 描述 |
|------|------|
| 唯一入口 | 聚合根是访问聚合内实体的唯一入口 |
| 边界保护 | 聚合外不能直接修改聚合内实体 |
| 事务边界 | 一个聚合 = 一个事务 |

### 值对象

| 规则 | 描述 |
|------|------|
| 不可变 | 值对象创建后不可修改 |
| 相等性 | 值对象通过属性值判断相等 |
| 自验证 | 值对象构造时验证自身有效性 |

### 领域服务

| 规则 | 描述 |
|------|------|
| 无状态 | 领域服务不应持有状态 |
| 跨实体 | 封装跨实体的业务逻辑 |
| 领域语言 | 方法名应使用领域语言 |

### 仓储模式

| 规则 | 描述 |
|------|------|
| 接口在领域层 | 仓储接口定义在领域层 |
| 实现在基础设施层 | 仓储实现在基础设施层 |
| 隐藏实现细节 | 不暴露数据库细节 |

## 输出契约

```yaml
# 输出：设计文档 + 审查报告
stage_id: "stage-1-understand-design"
version: "1.0.0"

postconditions:
  required_outputs:
    - name: "spec_document"
      type: file
      path: "specs/{name}-spec.md"
      format: "Markdown，包含 Gherkin 场景"
    - name: "architecture_document"
      type: file
      path: "docs/02_logical_workflow/{name}-architecture.md"
      format: "Markdown，包含架构图、接口定义"
    - name: "design_document"
      type: file
      path: "src/{module}/design.md"
      format: "Markdown，包含实现设计"
    - name: "design_review_report"
      type: json
      path: "contracts/stage-1-review.json"
      format: |
        {
          "review_status": "passed|failed",
          "reviewers": ["审查者列表"],
          "issues": ["问题清单"],
          "approvals": ["批准者签名"]
        }

invariants:
  - "设计必须符合 P0 级约束（质量红线）"
  - "设计必须符合 P1 级约束（架构原则）"
  - "设计文档必须包含可测试性设计"
  - "所有接口必须有明确的输入输出定义"
```

## 质量门控

| 检查项 | 通过条件 | 失败处理 |
|--------|----------|----------|
| 需求边界清晰 | 功能范围明确，无歧义 | 返回需求分析 |
| 技术方案对齐 | 技术选型合理，与现有系统兼容 | 返回架构设计 |
| 验收标准具体 | 每个功能都有明确的验收条件 | 返回需求分析 |
| 关键假设确认 | 所有关键假设已与用户确认 | 返回需求分析 |
| 架构图清晰 | 架构图完整，模块边界明确 | 返回架构设计 |
| 接口定义完整 | 所有接口都有明确的输入输出定义 | 返回实现设计 |
| 与现有系统无冲突 | 不与现有架构/接口冲突 | 返回架构设计 |
| 设计可行 | 技术方案可实现 | 返回实现设计 |

## 状态定义

| 状态 | 触发者 | 含义 | 后续动作 |
|------|--------|------|----------|
| `[STAGE_1_STARTED]` | 阶段 0 通过 | 阶段 1 开始 | 执行需求分析 |
| `[STAGE_1_ANALYZING]` | 需求分析 | 正在分析需求 | 等待分析完成 |
| `[STAGE_1_DESIGNING]` | 设计阶段 | 正在设计 | 等待设计完成 |
| `[STAGE_1_REVIEWING]` | 设计审查 | 正在审查设计 | 等待审查完成 |
| `[STAGE_1_WAITING_CONFIRM]` | 审查完成 | 等待用户确认 | 用户确认后进入阶段 2 |
| `[STAGE_1_PASSED]` | 用户确认 | 阶段 1 通过 | 进入阶段 2 |
| `[STAGE_1_FAILED]` | 审查失败 | 设计审查未通过 | 返回设计修正 |
