# 阶段 4：归档与演化

> **阶段目标**: 创建归档记录，评估升级，更新重规范

## 阶段概述

阶段 4 负责创建归档记录，评估是否需要升级为重规范，并更新 CHANGELOG。

## 归档记录规范

### 归档记录模板

```markdown

- 技术确认：[confirmation.md 链接]
- 完成时间：[YYYY-MM-DD]
- 实际耗时：[X 人天]
- 负责人：[姓名]

## 实现摘要
- [实现要点 1]
- [实现要点 2]
- [实现要点 3]

## 变更记录
- [变更 1：描述及原因]
- [变更 2：描述及原因]

## 升级评估
- **是否需要升级为重规范**：是/否
- **升级原因**：[原因描述]
- **目标文档**：
  - [ ] [文档 1](链接)
  - [ ] [文档 2](链接)

## 重规范更新内容
### 更新文档 1：[文档名称]
- 新增内容：[描述]
- 修改内容：[描述]

## 经验教训
- [经验 1]
- [经验 2]
- [改进建议]

归档人：[姓名]
归档时间：[YYYY-MM-DD]
```

## 升级评估规范

### 必须升级条件

| 条件 ID | 条件描述 | 升级目标 |
|---------|---------|---------|
| UPG-001 | 影响核心架构原则 | P0 级：架构原则文档 |
| UPG-002 | 新增安全要求或发现安全漏洞 | P0 级：安全基线文档 |
| UPG-003 | 新增核心业务流程 | P1 级：系统规范文档 |
| UPG-004 | 新增/修改核心 API | P2 级：API 契约文档 |
| UPG-005 | 新增/修改核心数据模型 | P2 级：数据模型文档 |
| UPG-006 | 新增核心领域概念 | P2 级：领域模型文档 |

### 可选升级条件

- 功能稳定运行超过 3 个月
- 多个轻规范涉及同一模块，可合并升级
- 团队决定需要长期维护的功能

## CHANGELOG 更新规范

### 变更类型

| 类型 | 描述 | 示例 |
|------|------|------|
| 新增（Added） | 新功能 | 新增用户登录功能 |
| 变更（Changed） | 功能变更 | 修改 API 响应格式 |
| 修复（Fixed） | Bug 修复 | 修复登录超时问题 |
| 弃用（Deprecated） | 即将移除 | 弃用旧版 API |
| 移除（Removed） | 已移除 | 移除废弃的接口 |
| 安全（Security） | 安全相关 | 修复安全漏洞 |

### CHANGELOG 格式

```markdown
## [版本号] - YYYY-MM-DD

### 新增
- [新增内容 1]
- [新增内容 2]

### 变更
- [变更内容 1]
- [变更内容 2]

### 修复
- [修复内容 1]

### 安全
- [安全相关内容]
```
