# 阶段 0：规范重量选择

> **阶段目标**: 评估需求复杂度，推荐规范重量（重规范/轻规范）

---

## 阶段概述

阶段 0 是工作流的入口，负责评估需求的复杂度，并推荐使用重规范还是轻规范路径。

---

## 复杂度评估标准

### 功能复杂度

| 分数 | 描述 | 示例 |
|------|------|------|
| 1-3 | 简单功能 | 配置修改、文案调整 |
| 4-6 | 中等功能 | 新增接口、数据表 |
| 7-10 | 复杂功能 | 新模块、新系统 |

### 技术复杂度

| 分数 | 描述 | 示例 |
|------|------|------|
| 1-3 | 无技术影响 | 纯前端修改 |
| 4-6 | 中等技术影响 | 新增依赖、接口变更 |
| 7-10 | 重大技术影响 | 架构变更、技术栈变更 |

### 影响范围

| 分数 | 描述 | 示例 |
|------|------|------|
| 1-3 | 单模块 | 单个服务/模块 |
| 4-6 | 多模块 | 2-5 个模块 |
| 7-10 | 跨系统 | 跨多个系统 |

### 综合复杂度计算

```
综合复杂度 = (功能复杂度 + 技术复杂度 + 影响范围) / 3

推荐规则：
- 综合复杂度 >= 7 → 重规范
- 综合复杂度 4-6 → 轻规范（可升级）
- 综合复杂度 <= 3 → 快速路径
```

## 规范重量选择标准

### 重规范

**触发条件**（任一满足）：
- ✅ 从 0 到 1 的核心系统开发
- ✅ 跨团队协作的大型功能
- ✅ 安全/金融等高风险模块
- ✅ 需要长期演进的基础设施
- ✅ 架构级变更
- ✅ 核心接口变更

**必需文档**：
- [ ] 工程宪章（4 个文档）
- [ ] 系统规范
- [ ] API 契约
- [ ] 数据模型
- [ ] 约束矩阵

### 轻规范

**触发条件**（全部满足）：
- ✅ 小功能 / 增量需求
- ✅ 不涉及架构变更
- ✅ 不涉及核心接口变更
- ✅ 影响范围可控

**必需文档**：
- [ ] proposal.md（需求提案）
- [ ] confirmation.md（技术确认）
- [ ] archive.md（归档记录）

### 快速路径

**触发条件**（全部满足）：
- ✅ 单文件修改
- ✅ 行数变化 < 30 行
- ✅ 无逻辑变更
- ✅ 无测试依赖
- ✅ 无文档依赖

**跳过阶段**：
- 跳过阶段 1（设计阶段）

## 质量门控

| 检查项 | 通过条件 | 失败处理 |
|--------|----------|----------|
| 需求描述清晰 | 长度 >= 10 字符，无歧义 | 返回需求澄清 |
| 复杂度评估合理 | 分数在 1-10 范围内 | 重新评估 |
| 推荐结果明确 | 明确推荐重规范/轻规范/快速路径 | 重新推荐 |
