# 阶段 2：实现与验证

> **阶段目标**: TDD 循环执行，约束验证，代码审查

---

## 阶段概述

阶段 2 负责将设计转换为代码，通过 TDD 循环确保质量，并进行约束验证和代码审查。

---

## 输入契约

```yaml
# 输入：设计文档 + 审查报告
preconditions:
  required_inputs:
    - name: "design_document"
      type: file
      path: "src/{module}/design.md"
      validation: "设计文档必须存在且通过审查"
    - name: "design_review_report"
      type: json
      path: "contracts/stage-1-review.json"
      validation: "审查状态必须为 passed"
    - name: "constraint_matrix"
      type: file
      path: "05_constraints/constraint-matrix.md"
      validation: "约束矩阵必须存在"
  constraints:
    - "设计文档必须通过审查"
    - "必须明确适用的 P0/P1 级约束"
```

## TDD 循环执行规范

### 红阶段（Red）

```
┌─────────────────────────────────────────┐
│  红阶段：编写失败测试                    │
│  - 根据规范中的 BDD 场景生成单元测试     │
│  - 验证测试因功能未实现而失败           │
│  - 测试代码必须符合 P3 级测试规范       │
└─────────────────────────────────────────┘
```

**任务序列**：

| 任务 | 规范依据 |
|------|---------|
| 生成测试代码 | BDD 场景（Gherkin） |
| 验证测试失败 | 测试规范（P3 级） |

### 绿阶段（Green）

```
┌─────────────────────────────────────────┐
│  绿阶段：编写实现                        │
│  - 编写满足测试的最小代码增量           │
│  - 确保测试通过                         │
│  - 代码必须符合 P2/P3 级规范           │
└─────────────────────────────────────────┘
```

**任务序列**：

| 任务 | 规范依据 |
|------|---------|
| 生成实现代码 | 模块规范（P2 级） |
| 验证测试通过 | 验收条件 |

### 重构阶段（Refactor）

```
┌─────────────────────────────────────────┐
│  重构阶段：优化代码                      │
│  - 在测试保护下优化代码结构             │
│  - 引入 DDD 战术模式（聚合根、值对象等）│
│  - 确保不引入新的失败                   │
└─────────────────────────────────────────┘
```

**任务序列**：

| 任务 | 规范依据 |
|------|---------|
| 优化结构 | DDD 战术模式 |
| 验证全部通过 | 回归测试 |

## 约束验证规范

### P0 级约束验证

| 约束 | 验证方法 | 通过条件 |
|------|----------|----------|
| 禁止硬编码密钥 | 静态分析 | 零违反 |
| 核心模块覆盖率 100% | 覆盖率工具 | ≥ 100% |
| 禁止强制解包 | 静态分析 | 零违反 |
| 禁止循环依赖 | 依赖分析 | 零违反 |

### P1 级约束验证

| 约束 | 验证方法 | 通过条件 |
|------|----------|----------|
| API 响应时间 < 500ms | 性能测试 | 平均 < 500ms |
| 优先使用项目已有库 | 依赖审查 | 无新依赖或已审批 |

### P2 级约束验证

| 约束 | 验证方法 | 通过条件 |
|------|----------|----------|
| 遵循命名约定 | 代码风格检查 | 无违规命名 |
| 公共 API 必须注释 | 文档检查 | 无缺失注释 |

```yaml
# 输出：代码变更 + 审查报告 + 测试报告
stage_id: "stage-2-implement-verify"
version: "1.0.0"

postconditions:
  required_outputs:
    - name: "code_changes"
      type: git_diff
      path: "git commit"
      format: "符合项目代码规范的代码变更"
    - name: "code_review_report"
      type: json
      path: "contracts/stage-2-code-review.json"
      format: |
        {
          "review_status": "passed|failed",
          "coverage": 0-100,
          "issues": ["问题清单"],
          "security_scan": "安全扫描结果"
        }
    - name: "test_report"
      type: json
      path: "contracts/stage-2-test-report.json"
      format: |
        {
          "test_status": "passed|failed",
          "total_tests": 数字,
          "passed_tests": 数字,
          "coverage": 0-100
        }
    - name: "constraint_report"
      type: json
      path: "contracts/stage-2-constraint-report.json"
      format: |
        {
          "p0_violations": 0,
          "p1_warnings": 数字,
          "p2_warnings": 数字
        }

invariants:
  - "代码必须通过 P0 级约束验证（零违反）"
  - "代码必须通过 P1 级约束验证（警告可接受）"
  - "核心模块测试覆盖率必须 >= 100%"
  - "禁止强制解包（unwrap/expect）"
  - "禁止硬编码密钥等敏感信息"
```

## 质量门控

| 检查项 | 通过条件 | 失败处理 |
|--------|----------|----------|
| 代码规范 | 通过 lint 检查 | 返回代码修正 |
| 测试通过 | 所有测试通过 | 返回测试修正 |
| 文档同步 | 文档与代码一致 | 返回文档修正 |
| P0 约束 | 零违反 | 返回代码修正 |
| P1 约束 | 无阻断性问题 | 返回代码修正 |
| 代码审查 | 审查通过 | 返回代码修正 |

## 状态定义

| 状态 | 触发者 | 含义 | 后续动作 |
|------|--------|------|----------|
| `[STAGE_2_STARTED]` | 阶段 1 通过 | 阶段 2 开始 | 执行 TDD 循环 |
| `[STAGE_2_TESTING]` | TDD 红阶段 | 正在编写测试 | 等待测试完成 |
| `[STAGE_2_IMPLEMENTING]` | TDD 绿阶段 | 正在实现代码 | 等待实现完成 |
| `[STAGE_2_REFACTORING]` | TDD 重构阶段 | 正在重构代码 | 等待重构完成 |
| `[STAGE_2_VERIFYING]` | 约束验证 | 正在验证约束 | 等待验证完成 |
| `[STAGE_2_REVIEWING]` | 代码审查 | 正在审查代码 | 等待审查完成 |
| `[STAGE_2_WAITING_CONFIRM]` | 审查完成 | 等待用户确认 | 用户确认后进入阶段 3 |
| `[STAGE_2_PASSED]` | 用户确认 | 阶段 2 通过 | 进入阶段 3 |
| `[STAGE_2_FAILED]` | 审查失败 | 代码审查未通过 | 返回代码修正 |
