## 质量门控机制

> 每个阶段完成后必须执行门控检查，确保质量：

| 阶段 | 门控检查项 | 通过条件 | 失败处理 |
|------|-----------|----------|----------|
| 阶段 0 | 需求描述清晰、复杂度评估合理 | 全部通过 | 返回需求澄清 |
| 阶段 1 | 设计文档完整、符合 P0 级约束、通过审查 | 全部通过 | 返回设计修正 |
| 阶段 2 | 代码规范、测试通过、约束验证通过 | 全部通过 | 返回实现修正 |
| 阶段 3 | 文档同步完成、索引更新正确 | 全部通过 | 返回同步修正 |
| 阶段 4 | 归档记录完整、CHANGELOG 更新 | 全部通过 | 返回归档修正 |

**门控失败约束**：
- 每次失败都需要用户决策
- 用户可选择：修复后重试、回滚、终止
- 不与三错即停机制关联

## 契约式协作

### 核心原则

每个环节（Stage）都是一个独立的**契约单元**，通过明确的输入输出契约进行协作：

```
┌─────────────────────────────────────────┐
│  环节 N                                  │
│                                         │
│  前置条件 (Precondition):               │
│  - 输入必须满足的条件                   │
│                                         │
│  处理逻辑:                              │
│  - 独立的上下文空间                     │
│  - 不依赖其他环节的内部状态             │
│                                         │
│  后置条件 (Postcondition):              │
│  - 输出必须满足的条件                   │
│  - 必须产生的产物                       │
│                                         │
│  不变式 (Invariant):                    │
│  - 环节内部必须保持的性质               │
└─────────────────────────────────────────┘
```

### 上下文隔离原则

**核心规则**：各环节拥有独立的上下文空间，不共用状态。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  环节 N-1    │    │   环节 N     │    │  环节 N+1    │
│              │    │              │    │              │
│  Context#1   │───▶│  Context#2   │───▶│  Context#3   │
│              │    │              │    │              │
│  输出契约    │    │  输入契约    │    │  输入契约    │
│  (JSON/YAML) │    │  (JSON/YAML) │    │  (JSON/YAML) │
└──────────────┘    └──────────────┘    └──────────────┘
     ↓                     ↓                     ↓
 归档到重规范          归档到重规范          归档到重规范
```

**上下文传递方式**：
1. **仅通过契约传递**：环节间只通过输入输出契约（JSON/YAML 文件）传递信息
2. **禁止共享内存**：不共用变量、状态、缓存
3. **禁止隐式依赖**：所有依赖必须显式声明在输入契约中
4. **上下文版本化**：每个上下文都有唯一 ID 和版本号

## 路径选择

### 重规范路径

**适用场景**：
- 从 0 到 1 的核心系统开发
- 跨团队协作的大型功能
- 安全/金融等高风险模块
- 需要长期演进的基础设施

**流程**：
```
阶段 0 → 阶段 1（完整设计）→ 阶段 2 → 阶段 3 → 阶段 4
```

**产出物**：
- 完整的工程宪章文档
- 完整的系统规范文档
- 完整的约束矩阵

### 轻规范路径

**适用场景**：
- 小功能 / 增量需求
- UI、接口、配置类改动
- 试验性功能
- Bug 修复
- 性能优化（不影响架构）

**流程**：
```
阶段 0 → 阶段 1（简化设计）→ 阶段 2 → 阶段 3 → 阶段 4
```

**产出物**：
- proposal.md（需求提案）
- confirmation.md（技术确认）
- archive.md（归档记录）

### 快速路径

**适用场景**：
- 单文件 + <30 行 + 无逻辑变更

**流程**：
```
阶段 2（跳过设计）→ 阶段 3
```
