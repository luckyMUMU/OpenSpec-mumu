# SOP 体系优化实施计划（基于 SDD 与约束驱动开发）

> **版本**: v3.0.0  
> **制定时间**: 2026-02-28  
> **核心原则**: **Spec-first（规范驱动 Skill）** + 契约是规范子集 + 升级归档机制

---

## 一、优化目标

### 1.0 核心理念：规范为核心，Skill 为实现方式

#### 第一性原理：规范是第一性产物

**核心转变**：
- ❌ 旧理念：Skill-first（Skill 是核心）
- ✅ 新理念：**Spec-first（规范是核心）**

**规范的定义**：
- 规范是系统的"宪法"，定义系统应该做什么、不应该做什么
- 规范是唯一真理源，代码、测试、文档都是规范的实现
- 规范驱动整个开发流程：规范 → 计划 → 任务 → 实现 → 验证

**Skill 的定位**：
- Skill 是执行规范的工具
- Skill 是规范的"翻译器"，将规范转换为代码、测试、文档
- Skill 的能力边界由规范定义

#### 规范驱动的 8 大核心原则

1. **准度 > 速度** - 规范质量优先于实现速度，规范不明确不实现
2. **规范先行** - 先写规范，后写代码；规范未通过审查不实现
3. **渐进披露** - 规范按 P0→P1→P2→P3 层级按需展开
4. **少即是多** - 规范简洁优先：先复用→改进→新建→清理
5. **验证独立** - 验证规范是否被满足，而非验证代码是否正确
6. **规范维度** - 按规范的作用范围划分：系统规范→模块规范→接口规范
7. **无依据不决断** - 规范中的每个约束都必须有依据（业务需求、安全要求等）
8. **规范须确认** - 所有规范必须通过用户/架构师明确确认

#### Skill 作为实现方式

**Skill 的分类**（按规范层级划分）：

1. **规范类 Skill**（定义规范）：
   - `sop-requirement-analyst`：需求分析 → 生成 P1/P2 级规范
   - `sop-architecture-design`：架构设计 → 生成 P1 级规范
   - `sop-implementation-designer`：实现设计 → 生成 P2/P3 级规范

2. **实现类 Skill**（执行规范）：
   - `sop-code-explorer`：探索代码 → 验证规范是否被遵循
   - `sop-code-implementation`：代码实现 → 将规范翻译为代码
   - `sop-test-implementation`：测试实现 → 将规范翻译为测试

3. **验证类 Skill**（验证规范）：
   - `sop-architecture-reviewer`：架构审查 → 验证实现是否符合 P1 级规范
   - `sop-code-review`：代码审查 → 验证实现是否符合 P2/P3 级规范

4. **编排类 Skill**（管理规范）：
   - `sop-workflow-orchestrator`：流程编排 → 管理规范的版本和演化
   - `sop-document-sync`：文档同步 → 确保规范与实现同步

**Skill 的工作方式**：
```
规范（输入） → Skill（翻译器） → 产物（输出）
   ↓                              ↓
 验证                           代码/测试/文档
```

### 1.1 优化目标（在规范为核心基础上的优化）

1. **建立重规范体系**：创建项目的持久化产出文档（工程宪章、系统规范、架构约束）
2. **引入轻规范流程**：对日常变更采用轻量级规范（提案→确认→实现→归档）
3. **优化 SOP 流程**：简化协作链条，明确各阶段输入输出，增加自动化验证
4. **实现规范演化**：轻规范完成后归档到重规范，保持规范的持续演进
5. **引入契约式设计**：各环节通过明确的输入输出契约进行协作，不共用上下文
6. **简化具体产物**：不强制要求完整的 17 个 Skill 产物，按需简化

### 1.3 预期收益

- **可控性提升**：核心规范（重规范）确保系统一致性
- **效率提升**：日常变更（轻规范）降低决策成本
- **可维护性提升**：规范持续演化，避免文档与代码脱节
- **AI 协作优化**：明确护栏，减少指令式交互
- **上下文隔离**：各环节拥有独立上下文，避免状态污染
- **契约保障**：前置条件、后置条件、不变式确保各环节质量
- **产物简化**：按需生成，少即是多
- **规范驱动**：规范是第一性产物，Skill 是实现工具

## 二、重规范体系建设（简化版）

### 2.0 产物简化原则

**核心原则**：少即是多，按需生成

#### 简化策略

1. **核心文档必需**：工程宪章 4 个文档必须创建（P0 级约束）
2. **系统规范按需**：根据项目复杂度选择创建
3. **轻规范优先**：日常变更使用轻规范，完成后归档
4. **渐进式完善**：先有框架，再逐步填充

#### 产物分级

| 分级 | 文档类型 | 必需性 | 创建时机 |
|------|----------|--------|----------|
| **P0 级** | 工程宪章（4 个） | 必须 | 项目启动时 |
| **P1 级** | 系统规范、约束矩阵 | 推荐 | 0→1 核心功能开发时 |
| **P2 级** | API 契约、数据模型 | 按需 | 有明确需求时 |
| **P3 级** | 领域模型、ADR | 可选 | 复杂业务逻辑时 |

### 2.2 重规范文档清单（简化版）

#### 阶段 1：工程宪章（必须创建 - P0 级）

**文档 1**: `project-charter.md` - 项目宪章
- 项目愿景与目标
- 核心价值主张
- 关键干系人
- 技术愿景

**文档 2**: `quality-redlines.md` - 质量红线（P0 级约束）
- 安全红线（禁止硬编码密钥、禁止使用漏洞库等）
- 质量红线（核心模块测试覆盖率 100%、禁止强制解包等）
- 合规要求（数据隐私保护、许可证合规等）
- 违反处理（自动拦截、必须修复）

**文档 3**: `architecture-principles.md` - 架构原则（P0 级约束）
- 分层架构原则（Controller → Service → Repository）
- 依赖方向原则（依赖倒置、禁止循环依赖）
- 模块边界原则（高内聚低耦合）
- 技术选型边界（优先使用项目已有库）

**文档 4**: `security-baseline.md` - 安全基线（P0 级约束）
- 身份验证要求
- 数据加密要求
- 输入验证要求
- 日志与审计要求

**验收标准**：4 个文档必须创建，内容可以简洁，但约束必须明确

#### 阶段 2：系统规范（按需创建 - P1 级）

**创建时机**：
- ✅ 从 0 到 1 的核心系统开发
- ✅ 跨团队协作的大型功能
- ✅ 需要长期演进的基础设施

**文档 5**: `system-spec.md` - 系统级规范（按需）
- 系统功能边界
- 核心业务流程
- 外部接口定义
- 性能指标（响应时间、吞吐量、可用性）

**文档 6**: `api-contracts/` - API 契约（有 API 时创建）
- REST API 规范（OpenAPI 3.0）
- GraphQL Schema（如使用 GraphQL）
- 错误码规范

**文档 7**: `data-models/` - 数据模型（有数据库时创建）
- 数据库 Schema
- 领域对象定义
- 数据流转图

**文档 8**: `domain-models/` - 领域模型（复杂业务时创建）
- 统一语言（Ubiquitous Language）
- 核心域、支撑域、通用域
- 聚合、实体、值对象定义

#### DDD 战术模式映射（P2 级规范）

在模块规范中，应明确 DDD 战术模式的应用规则：

| DDD 模式 | 规范要求 | 示例 |
|---------|---------|------|
| **聚合根** | 必须定义实体的唯一访问入口 | "Order 必须作为 OrderItem 的唯一访问入口" |
| **值对象** | 规定不可变性及逻辑封装 | "所有货币计算必须使用 Money 值对象" |
| **领域服务** | 定义跨实体的业务逻辑接口 | "余额扣减逻辑必须封装在 DomainService 中" |
| **仓储模式** | 规定基础设施层与领域层解耦 | "仓储接口定义在领域层，实现细节严禁泄露" |
| **领域事件** | 规定系统间的异步通信契约 | "订单创建成功后必须发布 OrderCreated 事件" |

#### 限界上下文模块化规格管理

**核心原则**：在复杂系统中，单一的大型规格文件会导致处理效率低下。应按照限界上下文划分规格目录。

**目录结构示例**：
```
sop/
├── 02_specifications/
│   ├── ordering/           # 订单限界上下文
│   │   ├── spec.md         # 订单规格
│   │   ├── api-contracts/
│   │   └── domain-models/
│   ├── payment/            # 支付限界上下文
│   │   ├── spec.md         # 支付规格
│   │   ├── api-contracts/
│   │   └── domain-models/
│   └── inventory/          # 库存限界上下文
│       ├── spec.md
│       ├── api-contracts/
│       └── domain-models/
```

**限界上下文规范要求**：
- 每个限界上下文拥有独立的规格文件（spec.md）
- 上下文映射明确跨上下文依赖
- 降低 AI 的上下文负载
- 减少不同功能变更之间的冲突风险

**验收标准**：根据项目复杂度选择，不强制要求完整

#### 阶段 3：约束矩阵（推荐创建 - P1 级）

**文档 9**: `constraint-matrix.md` - 约束分层矩阵
- P0 级约束（战略层，不可违背）
- P1 级约束（战术层，原则上遵守）
- P2 级约束（执行层，允许例外）
- 约束验证方法
- 例外流程

**文档 10**: `tool-dictionary.md` - 工具使用策略（按需）
- 工具引入流程（少即是多原则）
- 工具版本确认（5 个维度验证）
- 工具使用规范（优先级、替代方案）

**文档 11**: `command-dictionary.md` - 命令字典（按需）
- SOP 命令定义（/plan、/spec、/ulw 等）
- 命令使用场景
- 命令输出规范

**验收标准**：constraint-matrix.md 必须创建，其他按需

### 2.3 与现有 SOP 的融合

#### Spec-first 架构转变

**架构理念转变**：
- ❌ 旧架构：Skill-first（Skill 是核心执行单元）
- ✅ 新架构：**Spec-first（规范驱动 Skill）**

**规范驱动 Skill 的工作机制**：
```
┌─────────────────────────────────────────┐
│  规范（Spec）                            │
│  - 定义"做什么"（What）                  │
│  - 定义约束条件（Constraints）           │
│  - 定义验收标准（Acceptance Criteria）   │
└────────────┬────────────────────────────┘
             │ 驱动
             ▼
┌─────────────────────────────────────────┐
│  Skill（执行工具）                       │
│  - 读取规范作为输入                      │
│  - 执行规范定义的任务                    │
│  - 生成规范要求的产物                    │
│  - 验证规范约束是否满足                  │
└────────────┬────────────────────────────┘
             │ 产出
             ▼
┌─────────────────────────────────────────┐
│  产物（Deliverables）                    │
│  - 代码、测试、文档                      │
│  - 必须符合规范定义的验收标准            │
└─────────────────────────────────────────┘
```

**Skill 的重新定位**：
- Skill 是规范的**执行器**，而非流程的**驱动者**
- Skill 的输入是规范（Spec），输出是产物（Deliverables）
- Skill 的能力边界由规范定义，不能超出规范范围
- 多个 Skill 可以协作执行同一个规范

#### 保留现有 Skill 矩阵

**Skill 分类保留**：
- 17 个 Skill 分类保留（编排、需求设计、实现质量、路径测试）
- 每个 Skill 的契约化接口保留
- 但 Skill 的调用方式改变：由规范驱动，而非人工触发

#### 简化产物要求

**不再强制要求**：
- ❌ 完整的 17 个 Skill 产物
- ❌ 每个阶段都必须有完整的文档
- ❌ 所有项目都使用深度路径

**改为按需生成**：
- ✅ 快速路径：单文件 + <30 行 + 无逻辑变更 → 跳过需求架构
- ✅ 深度路径：复杂功能 → 完整的需求→架构→设计→实现
- ✅ TDD 路径：核心业务/复杂逻辑 → 增加测试设计与实现

#### TDD 执行流程（规范驱动）

**核心原则**：TDD 循环（红-绿-重构）由 AI 代理自主执行，但必须基于规范设定的边界。

**执行流程**：

```
┌─────────────────────────────────────────┐
│  红阶段：编写失败测试                    │
│  - 根据规范中的 BDD 场景生成单元测试     │
│  - 验证测试因功能未实现而失败           │
│  - 测试代码必须符合 P3 级测试规范       │
└────────────┬────────────────────────────┘
             │ 测试失败
             ▼
┌─────────────────────────────────────────┐
│  绿阶段：编写实现                        │
│  - 编写满足测试的最小代码增量           │
│  - 确保测试通过                         │
│  - 代码必须符合 P2/P3 级规范           │
└────────────┬────────────────────────────┘
             │ 测试通过
             ▼
┌─────────────────────────────────────────┐
│  重构阶段：优化代码                      │
│  - 在测试保护下优化代码结构             │
│  - 引入 DDD 战术模式（聚合根、值对象等）│
│  - 确保不引入新的失败                   │
└─────────────────────────────────────────┘
```

**TDD 任务序列规范**：

| 阶段 | 任务 | 规范依据 |
|------|------|---------|
| 红 | 生成测试代码 | BDD 场景（Gherkin） |
| 红 | 验证测试失败 | 测试规范（P3 级） |
| 绿 | 生成实现代码 | 模块规范（P2 级） |
| 绿 | 验证测试通过 | 验收条件 |
| 重构 | 优化结构 | DDD 战术模式 |
| 重构 | 验证全部通过 | 回归测试 |

> **说明**：TDD 循环应在每个功能增量中独立执行，确保代码始终有测试保护。

#### 渐进披露机制保留

**L1-L4 层级不变**：
- L1: 核心概念（50 行内）→ 快速理解
- L2: Skill 矩阵（200 行内）→ 操作指南
- L3: 工作流程（500 行内）→ 详细规范
- L4: 参考文档（无限制）→ 完整模板

**从入口到任意文档最短跳数 ≤ 3**

### 2.3 重规范管理规则

#### 版本控制
- 所有重规范文档纳入 Git 管理
- 版本号遵循 SemVer（主版本。次版本。修订号）
- 每次变更必须有变更日志（CHANGELOG）

#### 变更流程
```
提案（轻规范） → 审查（技术负责人） → 实现 → 验证 → 归档（更新重规范）
```

#### 审查机制
- P0 级约束变更：需要技术委员会审批
- P1 级约束变更：需要技术负责人审批
- P2 级约束变更：需要模块负责人审批

## 四、SOP 流程优化

### 4.0 契约式设计原则

#### 契约与规范的关系

**核心定义**：契约（Contract）是 P2 级模块规范的子集。

```
┌─────────────────────────────────────────┐
│  P2 级：模块规范（Module Specification） │
│  ┌───────────────────────────────────┐  │
│  │  API 契约（API Contract）         │  │
│  │  - 接口定义                       │  │
│  │  - 输入输出约束                   │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │  数据模型（Data Model）           │  │
│  │  - 数据结构定义                   │  │
│  │  - 数据约束                       │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │  领域模型（Domain Model）         │  │
│  │  - 业务规则                       │  │
│  │  - 业务约束                       │  │

**契约在规范体系中的定位**：

| 规范层级 | 契约类型 | 作用范围 | 示例 |
|---------|---------|---------|------|
| **P0 级** | 工程宪章契约 | 整个项目 | 质量红线、安全基线 |
| **P1 级** | 系统契约 | 跨模块 | 系统接口契约、外部集成契约 |
| **P2 级** | 模块契约 | 单模块 | API 契约、数据模型契约 |
| **P3 级** | 实现契约 | 代码级别 | 函数契约、类契约 |

**契约文件的存储位置**：
```
sop/
├── 02_specifications/         # P1/P2 级规范
│   ├── api-contracts/         # API 契约（P2 级）
│   │   ├── user-api.yaml      # 用户模块 API 契约
│   │   └── order-api.yaml     # 订单模块 API 契约
│   ├── data-models/           # 数据模型契约（P2 级）
│   └── domain-models/         # 领域模型契约（P2 级）
│
└── contracts/                 # 环节间协作契约（临时）
    ├── stage-0-decision.json  # 阶段 0 输出契约
    ├── stage-1-review.json    # 阶段 1 输出契约
    └── stage-2-test-report.json  # 阶段 2 输出契约
```

**契约的两层含义**：
1. **规范契约**（持久）：定义模块/系统的接口约束，存放在 `02_specifications/` 下
2. **协作契约**（临时）：定义环节间的输入输出，存放在 `contracts/` 下，完成后归档

#### 契约设计核心理念

每个环节（Stage）都是一个独立的**契约单元**，通过明确的输入输出契约进行协作：

```
┌─────────────────────────────────────────┐
│  环节 N                                  │
│                                         │
│  前置条件 (Precondition):               │
│  - 输入必须满足的条件                   │
│                                         │
│  处理逻辑:                              │
│  - 独立的上下文空间                     │
│  - 不依赖其他环节的内部状态             │
│                                         │
│  后置条件 (Postcondition):              │
│  - 输出必须满足的条件                   │
│  - 必须产生的产物                       │
│                                         │
│  不变式 (Invariant):                    │
│  - 环节内部必须保持的性质               │
└─────────────────────────────────────────┘
```

#### 上下文隔离原则

**核心规则**：各环节拥有独立的上下文空间，不共用状态。

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  环节 N-1    │    │   环节 N     │    │  环节 N+1    │
│              │    │              │    │              │
│  Context#1   │───▶│  Context#2   │───▶│  Context#3   │
│              │    │              │    │              │
│  输出契约    │    │  输入契约    │    │  输入契约    │
│  (JSON/YAML) │    │  (JSON/YAML) │    │  (JSON/YAML) │
└──────────────┘    └──────────────┘    └──────────────┘
     ↓                     ↓                     ↓
 归档到重规范          归档到重规范          归档到重规范
```

**上下文传递方式**：
1. **仅通过契约传递**：环节间只通过输入输出契约（JSON/YAML 文件）传递信息
2. **禁止共享内存**：不共用变量、状态、缓存
3. **禁止隐式依赖**：所有依赖必须显式声明在输入契约中
4. **上下文版本化**：每个上下文都有唯一 ID 和版本号

### 4.3 一键启动模式优化

#### 当前 /ulw 模式

```

#### 优化后 /ulw 模式

```

    - 创建需求提案

```

### 4.4 约束验证自动化

#### 约束验证集成到 CI/CD

```yaml
# .github/workflows/constraint-validation.yml
name: 约束验证

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  validate-constraints:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      # P0 级约束验证
      - name: 安全扫描
        run: npm run security-scan
        
      - name: 覆盖率检查
        run: npm run test:coverage -- --threshold=100
        
      - name: 架构约束检查
        run: npm run test:architecture
        
      # P1 级约束验证
      - name: 性能测试
        run: npm run test:performance
        
      - name: 接口契约验证
        run: npm run test:contract
        
      # P2 级约束验证
      - name: 代码风格检查
        run: npm run lint
        
      # 生成验证报告
      - name: 生成验证报告
        run: npm run generate-constraint-report
        
      # 上传验证报告
      - uses: actions/upload-artifact@v3
        with:
          name: constraint-validation-report
          path: constraint-report.md
```

## 六、风险与缓解措施

### 风险 1：重规范文档过于复杂

**风险描述**：重规范文档创建耗时过长，团队失去耐心。

**缓解措施**：
- 采用渐进式方式，先创建框架，再逐步完善
- 优先创建 P0 级约束文档（质量红线、安全基线）
- 提供文档模板和示例，降低创建难度

### 风险 2：轻规范流程被滥用

**风险描述**：所有需求都使用轻规范，导致系统一致性下降。

**缓解措施**：
- 明确轻规范适用场景（见 3.3 节）
- `/spec-weight` 命令提供规范重量推荐
- 技术负责人审查规范重量选择

### 风险 3：归档机制执行不到位

**风险描述**：轻规范完成后未及时归档，导致重规范与实现脱节。

**缓解措施**：
- 将归档作为完成定义（DoD）的一部分
- 自动化归档流程（/archive 命令）
- 定期检查归档执行情况

### 风险 4：约束验证影响开发效率

**风险描述**：过多的自动化验证导致 CI/CD 时间过长。

**缓解措施**：
- P0 级约束必须验证，P1/P2 级约束可异步验证
- 优化验证工具性能
- 使用增量验证（只验证变更部分）
