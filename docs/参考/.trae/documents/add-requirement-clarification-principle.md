# 计划：增加需求澄清准则

## 一、需求分析

### 1.1 当前SOP结构分析

**阶段 0（规范重量选择）**：
- 仅评估需求复杂度，推荐规范重量
- 缺少深入挖掘用户真实意图的机制

**阶段 1（理解与设计）**：
- 步骤 1：需求分析（BDD场景描述）
- 质量门控中有"关键假设确认"，但缺乏系统性的提问框架
- 没有"长期目标 vs 短期实现"的区分机制

**现有约束体系**：
- P0级：安全、质量、架构红线（不可违背）
- P1级：跨模块约束（警告可接受）
- P2级：模块约束
- P3级：实现约束

**现有Skill**：
- sop-requirement-analyst：需求分析（缺少系统性提问机制）

### 1.2 新准则需求

**核心要求**：
1. 在设计阶段对用户进行多轮次多维度的提问
2. 确认用户的真实意图
3. 明确长期目标与短期实现

**适用阶段**：阶段 1（理解与设计）的步骤 1（需求分析）

```markdown
### 步骤 1: 需求分析（多轮次多维度提问）

#### 1.1 第一轮提问：业务背景与目标

**目标**：理解需求的业务背景和核心目标

**提问维度**：
| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 业务背景 | "这个需求要解决什么业务问题？" | 理解问题本质 |
| 目标用户 | "谁会使用这个功能？他们的核心诉求是什么？" | 明确用户画像 |
| 成功标准 | "如何衡量这个功能的成功？" | 定义验收标准 |
| 紧急程度 | "这个需求的优先级如何？是否有截止日期？" | 评估优先级 |

#### 1.2 第二轮提问：长期目标与短期实现

**目标**：区分长期愿景与短期可交付成果

**提问维度**：
| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 长期愿景 | "这个功能最终要达到什么状态？" | 理解终极目标 |
| 短期目标 | "第一期需要交付哪些核心能力？" | 定义 MVP 范围 |
| 演进路径 | "从短期到长期，计划的演进路径是什么？" | 规划迭代路线 |
| 技术债务 | "短期实现是否可以接受技术债务？如何偿还？" | 评估技术风险 |

#### 1.3 第三轮提问：边界与约束

**目标**：明确功能边界和约束条件

**提问维度**：
| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 功能边界 | "这个功能不包含什么？" | 避免范围蔓延 |
| 技术约束 | "是否有技术栈、性能、安全方面的约束？" | 识别技术限制 |
| 兼容性 | "是否需要兼容现有系统？有哪些依赖？" | 评估集成影响 |
| 资源限制 | "团队规模、时间预算、技术能力如何？" | 评估可行性 |

#### 1.4 第四轮提问：风险与假设

**目标**：识别潜在风险和关键假设

**提问维度**：
| 维度 | 问题示例 | 目的 |
|------|----------|------|
| 风险识别 | "这个需求最大的风险是什么？" | 提前规避风险 |
| 关键假设 | "我们做了哪些假设？这些假设是否已验证？" | 暴露隐藏假设 |
| 失败场景 | "如果失败，可能的失败模式是什么？" | 设计降级方案 |
| 回滚策略 | "如果上线后出现问题，如何回滚？" | 制定应急预案 |

## 需求约束

### P1-REQ-001: 需求澄清完整性

| 属性 | 值 |
|------|-----|
| **描述** | 阶段 1 必须完成多轮次多维度提问，记录用户真实意图 |
| **验证方法** | 检查 `specs/{name}-clarification.md` 文档完整性 |
| **违反处理** | 警告，建议补充澄清 |
| **例外流程** | 快速路径可跳过（需审批） |

**必需维度**：
- 业务背景与目标
- 长期目标与短期实现
- 边界与约束
- 风险与假设

### P1-REQ-002: 长期短期区分

| 属性 | 值 |
|------|-----|
| **描述** | 需求必须明确区分长期目标与短期实现 |
| **验证方法** | 检查澄清记录中是否包含"长期愿景"和"短期目标" |
| **违反处理** | 警告，建议补充区分 |
| **例外流程** | 简单需求可例外（需审批） |
```

#### 变更 3：`04_skills/specification/requirement-analyst/SKILL.md`

**新增步骤**：

```markdown
### 步骤 1.5: 多轮次多维度提问

1. 执行 4 轮提问（业务背景、长期短期、边界约束、风险假设）
2. 记录用户回答到澄清文档
3. 识别不一致或模糊的点
4. 追问直到清晰

### 步骤 1.6: 用户确认

1. 整理澄清记录
2. 请用户确认澄清内容
3. 记录确认人和确认时间
```

required_outputs:

    guarantees:

## 四、影响评估

### 4.1 对现有流程的影响

| 影响范围 | 影响程度 | 说明 |
|----------|----------|------|
| 阶段 1 流程 | 中等 | 增加提问环节，延长需求分析时间 |
| 阶段 1 产出物 | 轻微 | 新增 `clarification.md` 文档 |
| P1 约束 | 轻微 | 新增 2 个约束项 |
| Skill 定义 | 轻微 | 新增 2 个步骤 |

### 4.2 对质量的影响

**正面影响**：
- 减少需求理解偏差
- 明确长期短期目标，避免过度设计或设计不足
- 提前识别风险和假设
- 提高用户满意度

**潜在风险**：
- 增加需求分析时间
- 用户可能不耐烦回答多轮问题

**缓解措施**：
- 提问应简洁高效，避免重复
- 可根据需求复杂度调整提问深度
- 快速路径可跳过详细提问
